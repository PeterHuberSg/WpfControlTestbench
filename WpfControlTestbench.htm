Introduction

WPF controls support an amazing range of functionality and are accordingly challenging to develop, Even more difficult is to test if the control behaves properly under the many scenarios possible. The WPF Control Testbench allows you doing this visually. It displays all important properties from FrameworkElement and Controls. By changing them during run time you can create all kind of combinations quickly.

The WPF Control Testbench makes the inner working of the controls transparent. It shows in a real time trace when which property was changed and how the host and your control interact for the layouting. MeasureOverride, ArrangeOverride and OnRender are difficult to debug with Visual Studio, because the switching between your application and the debugger window fires events which will change the behaviour you your control completely.

When writing your own control, it is imperative that it behaves similar like Microsoft's controls. But the problem is that Microsoft's WPF documentation is rather useless. To examine the behaviour of a Microsoft control, host it in the WPF Control Testbench. Do the same for your own control and then verify that yours behaves the same in every scenario like Microsoft's.

Overview WPF Control Testbench

The WPF Control Testbench is basically a Window with 3 parts:

1) In the upper part is a panel with custom properties, added by the developer and the dependency properties from FrameworkElement, added automatically. If the custom control inherits from Control, also those properties get added.

2) In the lower left corner is the custom control under test

3) In the lower right corner is the trace, showing every changing dependency property change and layouting. As an added benefit, it measures the time consumed for each activity in milliseconds.

 

Control Sizing and Positioning

It is difficult make a control content look correctly, because the available space depends on numerous factors, like space given by host, margin, alignment and more. A simple formula would look like this:

Required Width = LeftMargin + LeftBorder + LeftPadding + Content Width + RightPadding + RightBorder + RightMargin

Of course, required width is often different from the width that the host can provide. So what happens when there is too much space ? Too little space ?

What happens if the Width property get sets (notice that it is not part of the formula above) ? What happens if Width is undefined ?

What happens if HorizontalAlignment changes from left to center, right or stretch ?

What happens when the Font(Size) changes ?

Does the host or your control handle Margin ? Padding ?

Congratulation if you know the answers to all these questions. If not, sadly you will hardly find the answers in Microsoft's documentation. But as the developer of a control, you need to have a precise understanding how this works.

To get answers to this kind of questions, use the WPF Control Testbench, change some values and see how a Microsoft control reacts. Then implement the same behaviour for your control.

Since getting this right requires checking many scenarios and a pixel more or less might make a difference, the Testbench allows changing the width, alignments, etc. easily. It displays dashed lines for margin and padding which can be moved with the mouse. Even the total available space can be easily changed with a mouse drag of the splitter line

Control Layouting

Layouting is the process WPF uses to calculate the location and space used by a control. It has a 2 steps:

1) Measure: The host tells the control how much space is available. The control tells the host how much space it wants, which can be more or less.

2) Arrange: After having queried all children, the host tells the control how much space it should use. Again, the control tells the host how much space it was actually using, which might be different.

When positioning can be done with a Grid, StackPanel, etc., the developer does not need to write code for layouting. But if there are special requirements how the content of a control needs to be arranged, you might have to write the layouting yourself. That is a big challenge and the main reason why I wrote the WPF Control Testbench. It displays in the trace what is the space the host wanted to give and how much space your control requested. Seeing this data will help you solve many puzzling layout problems.
